### 2. Phase 1: 스탯 등급 시스템 구현

**목표:** 뽑은 스탯이 얼마나 좋은지 시각적으로 표시

**등급 기준 설계:**
```
SSS (95%↑): 💎 분홍색 - 최상급
SS  (85%↑): ⭐ 금색   - 매우 좋음
S   (70%↑): ✨ 하늘색 - 좋음
A   (50%↑): 🟢 녹색   - 보통 이상
B   (30%↑): 🟡 주황색 - 보통 이하
C   (30%↓): ⚪ 회색   - 낮음
```

**구현 내용:**
1. `TowerGrowthSystem.js`에 `calculateStatGrade()` 메서드 추가
   - 정규화된 값(0~1)으로 등급 계산
   - 등급, 색상, 이모지 반환

2. `_rollStatGainsWithGrades()` 메서드 추가
   - 기존 `_rollStatGains()` 호출 후 각 스탯에 등급 추가
   - 4개 스탯 각각 등급 정보 포함

3. `StarUpgradeManager.js`에서 등급 배지 렌더링
   - `stat-grade-badge` 요소 추가
   - 배경색으로 등급 구분
   - 이모지 + 등급 텍스트 표시

4. `star-upgrade.css`에 배지 스타일 추가

**결과:**
- 스탯 옆에 등급 배지가 표시되어 어떤 스탯이 좋은지 직관적으로 확인 가능

### 4. Phase 3: UI 파티클 시스템

**목표:** HTML 위에 오버레이되는 Canvas 파티클 효과

**기존 시스템 분석:**
- `ParticleSystem.js`: 게임 필드용 (WebGL2, 최대 2000개)
- UI용으로는 별도 시스템 필요 (Canvas 2D, 최대 500개)

**구현 내용:**
1. `UIParticleSystem.js` 클래스 생성
   - Canvas 2D 기반 경량 파티클 시스템
   - HTML 위에 오버레이 (z-index: 9999, pointer-events: none)

2. 4가지 파티클 타입:
   - `emitSparkles()`: 반짝임 (작은 원, 방사형)
   - `emitConfetti()`: 컨페티 (사각형, 회전, 중력)
   - `emitGlow()`: 글로우 (원형 확대, 그래디언트)
   - `emitBurst()`: 폭발 (방사형, 빠른 속도)

3. 물리 시뮬레이션:
   - 속도 + 중력
   - 회전 (컨페티)
   - 페이드아웃 (알파값 감소)
   - 생명 주기 관리

4. 렌더 루프:
   - `requestAnimationFrame`으로 60fps 유지
   - `update(dt)` → `render()` 순환

**통합:**
- 각인 카드 클릭 시: 반짝임 20개
- 리롤 버튼 클릭 시: 폭발 30개
- 승급 완료 시: 컨페티 100개 + 글로우 + 반짝임

**메모리 관리:**
- 승급 완료 시 `destroy()` 호출
- Canvas 제거, 애니메이션 중단
- 파티클 배열 비우기

### 10. 승급 비용 차감 시점 수정

**문제:** "승급 버튼 클릭 시 비용 차감이 없고, 확인 버튼에서 차감됨"

**기존 흐름:**
1. upgrade-points 클릭 → 승급 UI 진입 (비용 차감 ❌)
2. 각인 선택
3. upgrade-confirm-btn 클릭 → 비용 차감 ✅ + 승급 적용

**문제점:**
- 사용자가 UI에서 여러 번 들어갔다 나올 수 있음
- 비용 차감 없이 스탯을 미리 볼 수 있음
- 의도와 다름

**수정된 흐름:**
1. upgrade-points 클릭 → 즉시 비용 차감 ✅ + 승급 UI 진입
2. 각인 선택 (이미 결제 완료 상태)
3. upgrade-confirm-btn 클릭 → 중복 차감 없이 승급 적용

**구현:**
1. `UIController.js`의 pointsDisplay 클릭 핸들러:
   - 비용 체크 → `spendBoth()` 호출
   - 실패 시 토스트 + 리턴
   - 성공 시 재화 표시 업데이트 + UI 진입

2. `StarUpgradeManager.js`의 `completeStarUpgrade()`:
   - 비용 차감 코드 제거
   - 주석으로 "이미 차감됨" 명시

**결과:**
- 승급 진입 = 결제 완료
- 중복 차감 없음
- 더 명확한 흐름

- 슬롯머신 같은 긴장감
- "어떤 스탯이 나올까?" 기대감
- 게임스러운 연출

### 3. UI 파티클 시스템

**동작:**
- 각인 카드 클릭: 하늘색 반짝임 20개
- 리롤 버튼 클릭: 빨간색 폭발 30개

### 4. 승급 완료 축하 연출

## 🤔 결정 사항 & 논의

### 1. Canvas 파티클 vs 기존 ParticleSystem 확장

**선택지:**
- A) 기존 `ParticleSystem.js` 확장 (WebGL2)
- B) UI 전용 Canvas 2D 파티클 시스템 생성

**결정:** B) UI 전용 시스템

**이유:**
- 기존 ParticleSystem은 게임 필드용 (게임 좌표계)
- UI는 HTML 좌표계 (DOM 위치 기준)
- 최대 파티클 수 제한 (500 vs 2000)
- 간단한 2D 렌더링으로 충분
- 분리하면 UI 파티클 제어가 쉬움

### 2. 롤링 애니메이션 타이밍

**선택지:**
- A) 모든 스탯 동시에 롤링 → 동시에 정지
- B) 순차적으로 롤링 시작 → 순차적으로 정지

**결정:** B) 순차 롤링 (150ms 간격)

**이유:**
- 슬롯머신의 긴장감 재현
- 각 스탯에 주목하게 됨
- 동시 정지는 너무 단조로움
- 순차 정지가 더 극적

### 3. 카드 디자인: 독자적 vs 노드 스타일

**초기 계획:**
- 각인 카드만의 독자적 디자인
- 원형 + 설명 영역

**사용자 요청:**
- 스킬트리 노드와 같은 스타일 사용
- 디자인 일관성

**결정:** 노드 스타일 재사용

**이유:**
- UI 일관성 (같은 게임처럼 느껴짐)
- 코드 재사용 (CSS 중복 최소화)
- 이미 검증된 디자인
- 사용자가 노드를 각인으로 선택한다는 개념 명확화

### 4. 승급 비용 차감 시점

**선택지:**
- A) 진입 시 차감 (현재 변경됨)
- B) 확인 시 차감 (기존)

**결정:** A) 진입 시 차감

**이유:**
- 승급 진입 = 결제 완료
- 중복 차감 방지
- 무료로 스탯 미리보기 방지
- 더 명확한 의도

## 🐛 문제 & 해결

### 문제 1: 각인 카드 디자인이 노드와 다름

**증상:**
- 사용자가 "많이 차이난다" 피드백
- 실제 HTML을 보여줌 (upgrade-node-card 구조)

**원인:**
- LLM이 `.skill-node`의 원형 디자인을 참고
- 실제 upgrade-node-card는 사각형 카드였음
- 인라인 스타일 사용으로 CSS 파일에 없었음

**해결:**
1. 사용자가 제공한 실제 HTML 구조 분석
2. 인라인 스타일을 CSS 클래스로 변환
3. 완전히 동일한 스타일 적용
   - 박스 섀도우 3중 구조
   - 아이콘 배지 그래디언트
   - 텍스트 크기 및 간격

**교훈:**
- 실제 렌더링된 HTML을 확인하는 것이 중요
- 인라인 스타일은 CSS 검색으로 찾을 수 없음
- 사용자 피드백이 가장 정확한 정보

### 문제 2: UI 구조 복잡도

**증상:**
- 리롤 섹션, 각인 섹션이 별도로 분리
- 스크롤이 많이 필요
- 관련 기능이 멀리 떨어짐

**해결:**
1. 모든 것을 `stat-comparison-section` 안으로 통합
2. 구분선으로 시각적 분리 (`border-top: 3px solid`)
3. 불필요한 섹션 제거 (`imprint-section`)

**결과:**
- UI가 훨씬 컴팩트
- 정보 밀도 증가
- 스크롤 감소

### 문제 3: 버튼이 너무 많은 공간 차지

**증상:**
- 리롤 버튼 (제목 + 비용 + 버튼 + 횟수) = 4줄
- 선택지 추가 버튼 (버튼 + 횟수) = 2줄
- 총 6줄이 버튼으로 사용됨

**해결:**
1. 두 버튼을 한 줄로 좌우 배치
2. 버튼 내부에 정보 통합:
   - 상단: 아이콘 + 제목
   - 하단: 비용 + 횟수
3. 높이 75px (적당한 터치 영역)

**결과:**
- 6줄 → 1줄 (75px)
- 공간 85% 절약
- 정보는 모두 유지
- 더 게임스러운 디자인

**교훈:** 큰 작업은 작은 단계로 나누고, 각 단계를 완료하면서 진행하라.

### 디자인 일관성의 가치

각인 카드를 독자적으로 디자인했다가 노드 스타일로 변경:

**독자적 디자인:**
- 각인만의 특별함 강조
- 하지만 게임이 파편화된 느낌

**노드 스타일 재사용:**
- 전체 게임이 하나로 통일된 느낼
- 사용자가 "스킬트리 노드를 각인으로 선택한다"는 개념 직관적으로 이해
- 코드 재사용

**교훈:** 디자인 일관성은 단순히 예쁘게 보이는 것 이상의 가치가 있다. 시스템의 개념을 전달한다.

### Canvas vs CSS 애니메이션

**Canvas 파티클:**
- 완전한 자유도
- 복잡한 물리 시뮬레이션
- 성능 관리 필요

**CSS 애니메이션:**
- GPU 가속
- 선언적
- 브라우저 최적화

**결론:**
- 파티클: Canvas 2D
- 트랜지션/변형: CSS
- 둘을 조합하면 최고

**교훈:** 적재적소에 기술을 사용하라. 모든 것을 하나의 방식으로 하려 하지 마라.

### 사용자 피드백의 정확성

이번 세션에서:
1. "각인 카드가 노드와 많이 다르다" → 실제 HTML 제공
2. "모듈 정보 필요 없다" → 즉시 제거
3. "버튼을 한 줄로" → 공간 85% 절약
4. "스탯과 각인을 같이" → 논리적 흐름

**패턴:**
- 사용자가 실제로 보고 느낀 것 지적
- 구체적인 개선안 제시
- 빠른 수정
- 즉시 개선 체감

**교훈:** 사용자는 무엇이 필요한지 정확히 안다. 빠르게 듣고 빠르게 반영하라.

### 초기 애니메이션의 중요성

"승급 창 처음 열 때도 롤링 재생해줘요"

**효과:**
- 똑같은 기능이지만 진입이 훨씬 극적
- 사용자가 "뭔가 특별한 일이 시작된다" 느낌
- 단 2줄 추가로 큰 효과

**교훈:** 초기 진입 연출이 전체 경험을 크게 좌우한다.

### 정보 밀도 vs 간결함

**정보 많이:**
- 모듈 정보 (Damage, Trigger)
- 비용 설명 (제목 + 값)
- 횟수 설명 (라벨 + 값)

**간결하게:**
- 효과만 (모듈 제거)
- 비용만 (⚡ 20)
- 횟수만 (0회)

**결과:** 간결한 것이 훨씬 보기 좋음

**교훈:** "혹시 필요할까봐" 추가한 정보는 대부분 노이즈다. 꼭 필요한 것만 보여주라.
