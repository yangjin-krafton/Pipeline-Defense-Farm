# Devlog: 모바일 게임 HUD 디자인 시안

**Date:** 2026-02-17

## Summary

Pipeline Defense Farm의 모바일 버전 HUD 디자인 시안 작성. 640:1138 비율 고정, 하단 슬라이드업 메뉴, 소화기관 건강 상태 표시, 타워 공격 범위 인디케이터, 스킬 트리 업그레이드 시스템 구현.

---

## 작업 흐름

### 1. 초기 요청: 모바일 HUD 디자인

사용자가 게임 테마에 맞는 기본 HUD 디자인을 요청. 처음에는 모든 기능을 포함한 완전한 HUD를 제작:

- 상단: 자원 표시, 웨이브 정보, 메뉴 버튼
- 좌측: 소화기관 상태 패널
- 우측: 타워 선택 그리드
- 하단: 미니맵, 컨트롤 버튼, 오디오 컨트롤
- 중앙: 알림 시스템
- 모달: 식사 체크인

**문제점:** 너무 많은 기능을 구현하여 복잡했고, 비주얼 컨셉 확인용으로는 과도함.

### 2. 단순화: 비주얼 컨셉 중심으로

사용자 피드백에 따라 최소 코드로 비주얼 컨셉만 확인할 수 있도록 재작성:

- 좌우 슬라이드 메뉴 방식
- 정적 디자인 (JavaScript 없음)
- 타워 설치/업그레이드 UI 중심

**결정:** 동작 없이 순수 CSS만으로 레이아웃 확인

### 3. UI 변경: 하단 슬라이드업 방식

사용자가 "게임 화면 내 타워를 클릭하면 하단에서 상세 메뉴가 올라오는 방식"을 요청:

**변경 사항:**
- 좌우 슬라이드 메뉴 → 하단 슬라이드업 메뉴로 교체
- 게임 화면 최대화 (1018px → 1078px)
- 타워 클릭 시 하단 50% 영역에 상세 정보 표시

**이유:** 게임 화면을 최대한 활용하면서도 필요할 때만 정보를 보여주는 UX

### 4. 비율 유지 방식 개선

처음에는 `aspect-ratio` CSS만 사용했으나 제대로 작동하지 않음.

**시도 1:** `aspect-ratio: 640 / 1138` + `width: 100vw` + `height: 100vh`
- 결과: 비율이 깨짐

**시도 2:** 미디어 쿼리로 계산
- 결과: 복잡하고 정확하지 않음

**최종 해결책:** 640x1138 고정 크기 + JavaScript `transform: scale()`
```javascript
const scale = Math.min(
    containerWidth / 640,
    containerHeight / 1138
);
gameScreen.style.transform = `scale(${scale})`;
```

**장점:**
- 디자인 시 픽셀 단위로 작업 가능
- 실행 시 자동 스케일링
- 모든 화면 크기에서 완벽한 비율 유지

### 5. 상단바 개선: 소화기관 건강 상태

사용자가 웨이브 정보 대신 소화기관 건강 상태 표시를 요청:

**이모지 하트 시스템:**
- 🟡 위: 💛💛💛🖤🖤 (60%)
- 🟢 소장: 💚💚💚💚🖤 (80%)
- 🔴 대장: ❤️🖤🖤🖤🖤 (20%)

**디자인 철학:**
- 프로그레스 바보다 직관적
- 색상 매칭으로 한눈에 상태 파악
- 귀여운 비주얼로 게임 테마 강화

### 6. 타워 공격 범위 인디케이터

사용자가 타워에 원형 공격 범위 표시 요청:

**구현:**
- 점선 원형 테두리 (180px 직경)
- 중앙에서 바깥으로 페이드되는 그라데이션
- 기본: 은은한 펄스 애니메이션
- 호버: 명확한 범위 표시

**레이어링:**
- 범위 인디케이터 (z-index: 1)
- 타워 본체 (z-index: 2)

### 7. 스탯 UI: 게임 유닛 스타일

기존 2x2 그리드 방식에서 게임 유닛 정보창 스타일로 변경:

**변경 전:**
```
┌──────┬──────┐
│ 공격력 │ 속도 │
│  45  │ 1.2s│
├──────┼──────┤
│ 범위 │ 효과 │
│ 2타일│ +15%│
└──────┴──────┘
```

**변경 후:**
```
⚔️ 공격력    [████████░░] 45
⚡ 공격속도   [████████░░] 1.2s
📍 범위      [████░░░░░░] 2타일
✨ 분해효과   [██████░░░░] +15%
```

**이모지 선택:**
- ⚔️ 공격력 (직관적)
- ⚡ 속도 (빠름)
- 📍 범위 (위치)
- ✨ 특수효과 (반짝임)

### 8. 업그레이드: 스킬 트리 시스템

기존 리스트 방식에서 문명 게임 스타일의 가로 스크롤 스킬 트리로 변경:

**구조:**
```
[Lv1]─→[Lv2]─→[Lv3]─→[Lv4]─┬→[속도↑]
  ✓       ✓       ★     🍎  │
                            └→[범위↑]→[Lv5]
                               🍎      🦠
```

**노드 상태:**
- **Unlocked (초록)**: 이미 연구 완료 (Lv 1-2)
- **Current (금색)**: 현재 레벨, 펄스 글로우 효과 (Lv 3)
- **Available (핑크)**: 연구 가능, 비용 표시 (Lv 4, 속도↑)
- **Locked (회색)**: 선행 조건 미충족 (범위↑, Lv 5)

**연결선:**
- 실선: 잠금 해제됨
- 점선: 연구 가능
- 흐린 점선: 잠김

**인터랙션:**
- 가로 스크롤로 트리 탐색
- 노드 호버 시 확대 효과
- 클릭으로 업그레이드 (실제 기능은 미구현)

## 결정 사항 & 논의

### 1. 복잡도 vs 컨셉 확인

**초기 방향:** 모든 기능을 포함한 완전한 HUD
- 장점: 실제 게임에 바로 사용 가능
- 단점: 복잡하고 수정 어려움

**최종 방향:** 비주얼 컨셉만 확인
- 장점: 빠른 디자인 확인, 간단한 코드
- 단점: 실제 기능은 나중에 추가 필요

**결정:** 단계적 접근 (디자인 확인 → 기능 구현)

### 2. 메뉴 위치: 좌우 vs 하단

**좌우 슬라이드:**
- 장점: 전통적인 UI 패턴
- 단점: 게임 화면 가림

**하단 슬라이드업:**
- 장점: 게임 화면 최대화
- 단점: 타워 클릭 필요

**결정:** 하단 슬라이드업 (게임 화면 우선)

### 3. 건강 바: 프로그레스 vs 이모지

**프로그레스 바:**
- 장점: 정확한 수치 표현
- 단점: 딱딱한 느낌

**이모지 하트:**
- 장점: 직관적, 귀여움
- 단점: 5단계만 표현 가능

**결정:** 이모지 하트 (게임 테마 강화)

### 4. 스탯 표시: 그리드 vs 리스트

**2x2 그리드:**
- 장점: 컴팩트
- 단점: 시각화 부족

**리스트 + 프로그레스 바:**
- 장점: 수치 시각화, 게임 유닛 느낌
- 단점: 세로 공간 더 필요

**결정:** 리스트 (직관성 우선)

### 5. 업그레이드: 리스트 vs 스킬 트리

**리스트:**
- 장점: 간단, 직관적
- 단점: 관계성 표현 부족

**스킬 트리:**
- 장점: 선행 조건 명확, 전략적 선택
- 단점: 공간 많이 필요, 복잡

**결정:** 스킬 트리 (깊이감 제공)

## 문제 & 해결

### 문제 1: 비율이 유지되지 않음

**증상:** `aspect-ratio` CSS만으로는 `width: 100vw` + `height: 100vh`일 때 비율이 깨짐

**원인:** 두 축 모두 100%로 설정하면 aspect-ratio가 무시됨

**해결:**
1. 고정 크기 (640x1138) 디자인
2. JavaScript로 화면 크기 측정
3. 작은 비율 선택
4. `transform: scale()` 적용

**결과:** 완벽한 비율 유지 + 픽셀 단위 디자인 가능

### 문제 2: 스킬 트리 연결선 배치

**증상:** SVG 연결선이 노드 위치와 맞지 않음

**원인:** 노드는 `transform: translate(-50%, -50%)`로 중앙 정렬했으나, SVG 좌표는 절대값

**해결:**
- 노드 중심 좌표 계산 (left + 20px)
- SVG 선 좌표에 반영
- 시각적으로 검증

**결과:** 연결선이 노드 중심을 정확히 연결

### 문제 3: 공격 범위가 타워 위에 표시됨

**증상:** 범위 인디케이터가 타워 이모지를 가림

**원인:** z-index 설정 없음

**해결:**
- 범위: `z-index: 1`
- 타워: `z-index: 2`
- 레이어링 명확화

**결과:** 타워가 항상 위에 표시

## 기술 노트

### CSS Transform Scale

**장점:**
- GPU 가속
- 리플로우 없음
- 부드러운 스케일링

**주의점:**
- `transform-origin` 설정 필요
- 이벤트 좌표는 원본 기준

### SVG in HTML

**사용 이유:**
- 연결선을 CSS만으로 그리기 어려움
- SVG는 벡터 그래픽으로 스케일에 강함

**구현:**
- `<svg>` 요소를 절대 위치
- `<line>` 요소로 연결선
- CSS 클래스로 스타일 지정

### Emoji Font Rendering

**문제점:**
- 브라우저/OS마다 다른 렌더링
- 크기가 일정하지 않음

**해결:**
- `filter: drop-shadow()` 로 글로우 효과
- `letter-spacing: -1px` 로 간격 조정
- `line-height: 1` 로 높이 통일

## 메모 / 인사이트

### 디자인 우선 접근

이번 세션에서는 기능 구현보다 **비주얼 컨셉 확인**에 집중했다. 실제 게임 로직 없이 순수 디자인만으로도:
- UI 레이아웃 검증
- 색상/폰트/간격 조정
- 사용자 피드백 수집

이는 나중에 기능을 구현할 때 방향성을 제시한다.

### 단일 파일의 힘

복잡한 프로젝트 구조 대신 **단일 HTML 파일**로 작업하니:
- 빠른 실행 (더블클릭)
- 쉬운 공유 (파일 하나)
- 간단한 수정

프로토타입 단계에서는 단일 파일이 훨씬 효율적이다.

### Scale Transform의 마법

비율 유지를 위해 복잡한 CSS 계산 대신 **고정 크기 + scale**을 사용하니:
- 디자인이 훨씬 쉬워짐 (픽셀 단위)
- 모든 화면에서 동일한 비율
- 코드가 단순해짐

때로는 JavaScript 한 줄이 CSS 100줄보다 낫다.

### 이모지 > 아이콘

아이콘 파일을 따로 만들지 않고 **이모지**를 사용하니:
- 외부 리소스 불필요
- 크로스 플랫폼 호환성
- 빠른 디자인 반복

단, 렌더링 일관성 문제는 있지만 프로토타입에는 충분하다.

### 문명 게임에서 배우기

업그레이드 UI를 문명 게임의 **기술 트리**에서 영감을 받아 만들었다:
- 선행 조건 시각화
- 전략적 선택 강조
- 진행 상황 한눈에 파악

좋은 게임의 UI는 다른 게임에서도 배울 점이 많다.
