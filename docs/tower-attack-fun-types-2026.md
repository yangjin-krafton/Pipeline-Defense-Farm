# 2026 기준 타워디펜스 타워 설계 마스터 문서 (큰그림 -> 세부그림)

## 문서 목적

- 전 타워를 같은 기준으로 설계해, 업그레이드 시 컨셉 중복/충돌을 방지
- `모듈 재사용`으로 구현 비용을 줄이고 밸런스 조정 단위를 통일
- 먼저 `큰 게임플레이 분기`를 확정하고, 이후 개별 타워 세부 노드를 설계

## 0) 공통 설계 원칙 (충돌 방지)

- 한 타워는 1개의 `주 역할` + 1개의 `보조 역할`만 가진다.
- 다른 분류의 시그니처를 빼앗지 않는다.
  - 예: 단일 화력은 광역 섬멸 성능을 가져가면 안 됨.
  - 예: 군중 제어는 단독 보스 삭제 성능을 가져가면 안 됨.
- 업그레이드 포인트는 타워당 `5포인트`, 노드는 `10~15개`.
- 노드 위치는 공통으로 `분기(1~3) / 중간(4~9) / 끝(10~12)`를 사용.
- 각 구간마다 강선택 2개 이상을 제공한다.

## 1) 모듈 재사용 사전 (전 타워 공통)

| 코드 | 모듈명 | 담당 기능 | 재사용 예시 |
|---|---|---|---|
| `TM` | TargetingModule | 타겟 우선순위/탐색 규칙 | 보스 우선, 체크포인트 우선 |
| `PM` | ProjectileModule | 투사체 형태/관통/속도 | 레일, 포탄, 체인, 장판 |
| `DM` | DamageModule | 기본 피해/계수/치명 | 단일 고점, 연쇄 피해 |
| `TB` | TagBonusModule | 음식 태그 상성 | 지방 추가효과, 유제품 특화 |
| `SM` | StatusModule | 상태이상 부여 | 감속, 방깎, 스턴, 산성 |
| `TR` | TriggerModule | 조건부 발동 | 처치 시, 체력 임계치 시 |
| `RM` | ResourceModule | 자원 획득/환급 | 영양 +1, 에너지 환급 |
| `SF` | SafetyModule | 리스크 완화 | 과열 완화, 과산 완화 |
| `AM` | AuraModule | 오라/범위 버프 | 사거리 오라, 공속 오라 |
| `UM` | UnitModule | 소환/유닛 제어 | 미니언 생산, 드론 AI |

## 3) 중복/충돌 체크리스트 (세부 설계 전 필수)

| 체크 항목 | 질문 | 통과 기준 |
|---|---|---|
| 역할 침범 | 이 노드가 타 분류 시그니처를 훔치나? | 훔치면 수치 50% 이하 또는 삭제 |
| 모듈 중복 | 같은 타워 내 동일 모듈만 반복되나? | 5포인트 빌드에서 최소 3종 모듈 사용 |
| 카운터 상실 | 모든 상황에 만능인가? | 최소 1개 명확한 약점 유지 |
| 음식 상성 | 탄수/지방/단백/유제품/탄산 대응이 분기되나? | 분기 노드에서 최소 2개 상성 선택 제공 |
| 리스크 관리 | 고점 빌드와 안정 빌드가 분리되나? | 끝 노드에서 폭딜 vs 안정 선택 제공 |

## 4) 세부 그림: 단일 화력 3종 (모듈 조립형)

### 공통 규칙

- 포인트: `5`
- 노드: `12`
- 위치: `분기 1~3 / 중간 4~9 / 끝 10~12`
- 위치별 강선택:
  - 분기: 즉시 고점 vs 운영 안정
  - 중간: 상성 대응 vs 리스크 완화
  - 끝: 피니시 폭딜 vs 경제/안정 마감

## 5) 구현 순서 (권장)

1. `모듈 시스템` 먼저 구현 (`TM/PM/DM/...`).
2. `전 타워 큰 분기 표`를 데이터 파일로 고정.
3. 타워별 노드는 모듈 번들만 선언.
4. 밸런스는 노드 수치가 아니라 모듈 계수부터 조정.
5. 새 타워 추가 시, 먼저 분류/분기/침범 금지선을 표에 등록 후 세부 설계.

## 6) 음식(Enemy) 속성 스키마: 모듈 식별/처리 기준

### 필수 속성 (모든 음식 공통)

| 필드 | 타입 | 설명 | 참조 모듈 |
|---|---|---|---|
| `id` | string | 음식 고유 식별자 | 전체 |
| `name` | string | 표시 이름 | UI/로그 |
| `tags` | string[] | 음식 태그 목록 (`carb`, `fat`, `protein`, `dairy`, `spicy`, `soda`, `fermented`) | `TB`, `TM`, `SM` |
| `hp` | number | 기본 체력(소화 난이도) | `DM`, `TR` |
| `speed` | number | 이동 속도 | `SM`, `TM` |
| `armor` | number | 물리/일반 피해 감소치 | `DM`, `SM` |
| `resist` | object | 상태저항 (`acid`, `slow`, `stun`, `mark`) | `SM`, `TB` |
| `size` | string | 충돌 크기 (`small`, `medium`, `large`) | `PM`, `TM` |
| `threat` | number | 위협도(우선 타겟팅 용도) | `TM` |
| `traits` | string[] | 특수 특성 (`boss`, `elite`, `split`, `shielded`) | `TM`, `TR`, `DM` |

### 권장 속성 (상성/리스크 정밀화)

| 필드 | 타입 | 설명 | 참조 모듈 |
|---|---|---|---|
| `digestStageBias` | object | 기관별 취약/강점 (`stomach`, `smallIntestine`, `largeIntestine`) | `TB`, `TR` |
| `troubleOnLeak` | object | 누수 시 트러블 기여도 (`gas`, `inflammation`, `constipation`) | `RM`, 시스템 |
| `acidLoad` | number | 산성 부담치(과산 시스템 연동) | `SF`, `SM` |
| `greaseLoad` | number | 지방 정체 유발치 | `SM`, `TB` |
| `fermentValue` | number | 발효/가스 누적치 | `SM`, `TR` |
| `nutritionValue` | number | 처치 보상 기본 영양 | `RM` |

### 모듈별 최소 참조 계약

| 모듈 | 반드시 읽는 음식 필드 | 처리 목적 |
|---|---|---|
| `TM` | `threat`, `traits`, `speed`, `tags` | 우선순위 타겟 선택 |
| `PM` | `size`, `speed` | 탄속/관통/명중 계산 |
| `DM` | `hp`, `armor`, `traits` | 최종 피해 계산 |
| `TB` | `tags`, `digestStageBias` | 음식 상성 보정 |
| `SM` | `resist`, `tags`, `greaseLoad`, `fermentValue` | 상태 부여/저항 판정 |
| `TR` | `hp`, `traits`, `tags` | 조건 발동(임계치/처치시) |
| `RM` | `nutritionValue`, `traits` | 영양/에너지 환급 |
| `SF` | `acidLoad`, `traits` | 과열/과산 리스크 완화 연산 |

### 검증 규칙 (데이터 입력 시)

- `tags`는 최소 1개 이상, 미등록 태그 금지
- `hp`, `speed`, `nutritionValue`는 0 초과
- `resist` 값은 `0.0 ~ 0.9` 범위
- `boss` trait이면 `threat >= 90` 권장
- `soda` 태그가 있으면 `fermentValue` 또는 `troubleOnLeak.gas`가 1 이상
